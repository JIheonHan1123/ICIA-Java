 토큰인증 cf) 토큰: 내 인증 정보를 담고있는 데이터
 - 로그인 인증은 로그인 한 사이트에서만 유효함 
 - 토큰 인증은 로그인처럼 보이지만 내부구현은 다르다
 		ex) 네이버에서 로그인하면 네이버페이, 치지직 등등의 사이트에서도 유효
 - 작동순서
 		1) 네이버에 로그인하면 네이버가 나(사용자)에게 Token을 발급해준다
   		2) 내가 네이버페이 쇼핑사이트에서 결제를 할 경우 내가 가진 토큰이 쇼핑사이트에 전송된다
   		3) 쇼핑사이트는 받은 토큰의 내용이 정확한지 네이버에게 확인받는다
   		4) 확인이 성공하면 결제가 이루어진다
 - 토큰인증 사용의 장점
 		보안을 내가 아닌 네이버에 의존한다 -> 신뢰성이 높고 빠른 조치가 가능
 - 토큰인증 사용의 단점
 		토큰을 안전하게 잘 관리해야하는데 "네이버는 믿을 수 있음. But 사용자를 어떻게 믿지?" 하는 문제가 생김
   		=> 그래서 토큰은 수명이 매우 짧다 (보통 30분 내외) -> 토큰 재발급 절차가 필요
   		=> 이런 토큰관리, 토큰재발급등을 규정한 규약. 토큰국제표준을 OAuth라고 한다.


 Access Modifier (접근 제어자)
 - public 		누구나 접근 가능
 - default		같은 패키지인 경우 접근 가능 -> 보통 권장X
 - protected	자식이면 접근 가능
 - private		아무도 접근 X
 - ex)  
 	 Class Sample {
		public int b;
		int a;						// default
		protected int c;
		private int d;
  	}
 
 
  Maven
  - 의존(dependency)
  		객체 A가 객체 B를 사용할 때 "의존"한다고 한다
  		ex) 사람은 스마트폰을 사용한다 -> 사람은 스마트폰에 의존한다
  - 내가 만든 프로젝트 = 내가 작성한 코드 + 의존하는 라이브러리
  - 라이브러리를 관리해주는 프로그램: maven 또는 gradle
     	이런 프로그램들을 "빌드도구" 또는 "의존성관리도구"라고 한다.
